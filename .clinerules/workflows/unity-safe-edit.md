# Workflow: Safe Unity Edit

破壊的な Unity 操作（スクリプト書き換え・プレハブ改変・シーン構造変更・削除）を
行うときの標準フロー。三層セーフティ（モデル / Cline / Git）を必ず経由する。

## 手順

1. **[Plan] スコープ確定**
   - 何を達成するタスクか、対象（どのシーン/プレハブ/スクリプト）、
     影響範囲、リスクを 3〜6 行で提示する。
   - 中規模チャンクに分解し、最初に着手するチャンクを1つ示す。
   - ここでユーザーの承認を待つ。承認なしに次へ進まない。

2. **[Git] 退避確認**
   - 作業ツリーがクリーンかユーザーに確認を促す。
   - 必要なら退避用コミット案を提示：
     `git add -A && git commit -m "before cline: <タスク名>"`

3. **[Act] 1チャンク実行**
   - 合意したチャンクだけを実行する。スコープを自己拡大しない。
   - 破壊的ツール実行の直前に「何を・どう変えるか」を1行宣言する。

4. **[確認] 結果検証**
   - Unity コンソール / コンパイルエラーを確認する。
   - 変更差分の要点を報告する。

5. **[判断] 続行 or 戻す**
   - 問題なければ次のチャンクへ（手順3へループ）。
   - 問題があれば即座に戻す案を提示：`git restore .` / `git reset --hard HEAD`。

6. **[締め] コミット提案**
   - まとまったら Conventional Commits 形式のコミットメッセージを提案する。

## 中断条件
- 連続実行が想定以上に伸びたら（Max Requests 接近）一旦止まって状況を報告する。
- 対象が曖昧になったら手を止めて1つ確認質問を投げる。
