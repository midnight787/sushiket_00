# Core Safety Rules — YANARI / Unity Project

## Objective
このプロジェクトは Unity 6.3 LTS を UnityMCP 経由で操作する。UnityMCP は
シーン・プレハブ・スクリプト・マテリアルを **直接かつ破壊的に** 書き換える権限を持つ。
このルールの目的は、AIエージェントの「考えすぎによる脱線」と「権限の過剰行使」を
構造的に防ぎ、人間の目視承認と Git 安全網を常に経由させることにある。

## 鉄則（最優先・例外なし）

1. **Plan で合意するまで Act に入らない。**
   - 破壊的操作（シーン変更・スクリプト書き換え・アセット削除・プレハブ改変）を
     伴うタスクは、必ず Plan mode で設計・影響範囲・対象ファイルを提示し、
     ユーザーの明示承認を得てから Act mode に移ること。
   - 「ついでに」「ついでなので」「あわせて」での作業範囲の自己拡大を禁止する。
     合意したスコープだけを実行する。

2. **1タスク = 1スコープ。**
   - 大きい要求は、Plan mode で中規模のチャンクに分解して提示する。
   - 各チャンクは「構造が先に定義され」「能力基準が明確」な単位にする。
   - 1チャンク実行 → 結果確認 → 次、のループを守る。複数チャンクを
     連続で無確認実行しない。

3. **不明点は手を動かす前に質問する。**
   - 対象が曖昧（どのシーン・どのプレハブ・どのスクリプトか不明）なまま
     破壊的操作をしない。1つだけ確認質問を投げる。

## reasoning effort 運用方針
- Plan mode: じっくり設計してよい（high 相当）。影響範囲とリスクを必ず明示する。
- Act mode: 合意済みの実装を手数優先で消化する。余計なリファクタ・余計な最適化・
  余計な「改善提案の実行」をしない。求められていない変更を加えない。
- `xhigh` 相当の最大推論はこのプロジェクトでは使わない（脱線リスクのため）。

## 禁止事項
- YOLO Mode 前提の挙動を取らない。全確認をスキップする運用を要求・推奨しない。
- ワークスペース外（親フォルダ・`.git`・ホームディレクトリ等）のファイルを
  読み書きしない。必要なら必ず明示承認を求める。
- ブラウザ操作・外部送信を勝手に行わない。
